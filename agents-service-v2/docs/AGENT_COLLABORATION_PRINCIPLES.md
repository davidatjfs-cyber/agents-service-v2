# Master / Planner / Agent 协作原则（不变更的大格局）

本仓库近期对 **BI 异常、任务催办、周度评分** 等做了「细节增强」，原则如下：

## 1. 角色关系（保持）

- **Master（路由与编排）**：仍负责意图识别、选 Agent、复杂流程的入口；不因 BI 管线而绕过 Master 主路径。
- **Planner（分析与计划）**：仍通过 `master-planner.js` 的 `planAndExecute` 生成分析与建议；BI 异常在落库后 **额外** 调用同一套 Planner，属于「异常场景的附加分析」，不是替代 Master。
- **各业务 Agent（data_auditor、OP 等）**：飞书问答、数据拉取、确定性回复等原有职责不变。

## 2. BI 异常管线（`anomaly-notify-pipeline.js`）的定位

- **仅**在 `anomaly_triggers` 写入之后执行：**即时通知责任人 → 建 `master_tasks` → 调用现有 `planAndExecute` → 再以营运督导口吻发跟进文案**。
- 不修改 Master 的常规对话路由规则；不替换 Planner 的核心实现，只传入合成问题与上下文。

## 3. 数据与「智能」边界

- **桌访产品异常**：与聊天侧共用 `fetchMergedTableVisitEntries` 合并数据源，但 **BI 检测与问答不同口径**：**仅**从桌访表独立列 **「今天不满意菜品」**（及「今天 不满意菜品」空格变体）拆菜名计数；**不**用「今天不满意的菜品」等其它列、**不**用 LLM 从自由文本猜菜名。周频任务 **每周一 05:00（上海）** 只跑 **上周一～上周日**（`shanghaiLastCompletedWeekBounds`）。  
**周度绩效扣分（出品经理）**：按**每个菜品**单独判断——该自然周内同一产品 ≥4 次记 10 分、≥2 次记 5 分（高档覆盖低档）；**多个产品同时触线则扣分相加**。Planner/数据说明须明确「今天不满意菜品」列与上周窗口，禁止与聊天侧宽口径混讲。

## 4. 若需调整协作格局

任何会改变「谁决策、谁执行、谁回用户」的改动，应单独评审，并在本文档更新说明。
