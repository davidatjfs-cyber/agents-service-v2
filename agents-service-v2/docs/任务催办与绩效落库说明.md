# 任务催办与工作态度备案 — 逻辑说明

## 两条独立链路

| 链路 | 适用状态 | 何时打标 `hr_performance_recorded`（语义：已产生工作态度记录） |
|------|-----------|------------------------------------------------------------------|
| **催办** `task-card-reminders.js` | 仅 **`pending_response`**，且 `source` 为随机抽检/定时巡检/BI 异常/自动协作 | 约每 **1 小时**可发 1 次催办；**满 3 次**后，再经过 **1 小时**仍无有效闭环 → **`hr_performance_recorded=true`**，`resolution_code=hr_attitude_unclosed_after_3_reminders`；**不写入 `agent_scores`** |
| **审核** `feishu-client.js` · `reviewTaskReply` | 用户回复任务卡后进入审核流程 | 回复 **连续 3 次审核不合格** → 同上打标，`resolution_code=hr_attitude_review_3x_fail`；**不写入 `agent_scores`**；**审核通过** → 任务应 **`resolved` 闭环** |

说明：列名仍为 `hr_performance_recorded` 系历史兼容，含义已调整为「已产生与工作态度相关的系统备案」，不再表示「周度绩效扣分已落库」。

## 与 HRMS 的衔接

- 总部群：`push_config.hq_group_chat_id` 会收到**工作态度备案**说明（非扣分）。
- 档案页「公司通知」：`hrms_user_notifications`，`type=attitude_record`，标题「工作态度备案」。
- **工作态度等级**（`new-scoring-model.js` · `getIncompleteTaskCount`）：当月 `master_tasks` 中，同一责任人满足 **催办≥3** 或 **审核≥3 次且仍未通过** 的任务数，按 `task_id` **去重**后计入。

## 已修复的代码问题（历史）

1. **审核通过后仍为 `pending_review`** → 已修复为 `resolved`。
2. **审核驳回后因有 `response_text` 导致不再催办** → 已删除该跳过逻辑。
3. **曾依赖 `agent_scores` 扣分** → 已取消；仅保留态度统计与通知备案。

## 定时任务

- 催办：`*/10` 分钟（上海时区），以 `dispatched_at` / `remind_count` / `last_reminder_at` 与当前时间比较为准。
