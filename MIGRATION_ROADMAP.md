## 2. 路由重叠矩阵

```
旧服务独有的功能域（152 条，不重叠）：
├── HR 管理 (~40 条)     → 员工、考勤、薪资、请假、入职、审批流
├── 运营管理 (~50 条)    → 日报、营收预测、毛利率、库存、销售额上传
├── 行政考核 (~40 条)    → 培训、考试、积分、促销、巡检
└── 基础设施 (~22 条)   → 文件上传、知识库、RAG 查询、多维表格同步

两个服务共享（6 条）：
├── POST /api/login         → 旧服务可能代理到新服务
├── POST /api/auth/heartbeat → 会话保活
├── GET  /                   → 管理后台首页
├── GET|POST /api/knowledge  → 知识库（各管各的表）
└── POST /api/webhook/feishu → 飞书 webhook

新服务独有的功能域（74 条，不重叠）：
├── AI Agent 系统      → 聊天、分派、处理
├── 异常检测           → 食安、BI、毛利率
├── KPI 评分           → 周度评分、月度考核、奖金计算
├── 自动化定时任务     → 晨报、执行力评、日报完成率、巡检
├── 配置管理           → Config CRUD、KPI 目标、SLA
└── LLM & 基础设施     → 模型路由、Embedding、队列
```

**结论**：两个服务几乎在完全不同的业务域运行，不是"功能重叠"，而是"互补"。

**旧服务的定时任务**（通过代码搜索）：

## 5. 迁移决策建议

### 核心结论：不建议迁移

理由：

1. **业务域完全不同** — 152 条独有路由对应的是完整的 HR 管理系统，迁移到 AI Agent 服务是错误的方向
2. **架构分离有明确分工** — 旧服务查考勤/薪资/审批，新服务跑 AI 分析/异常检测/自动推送，已经在功能上互补
3. **迁移成本极高** — 旧服务 51,380 行代码、单文件 19,366 行、依赖 6+ 个外部服务（OSS, COS, 飞书, 数据库），迁移 ≈ 重写
4. **现有架构已稳定运行** — 3 个服务在 ECS 零崩溃运行，旧服务 CPU 0%、内存 242MB，资源消耗可控

### 替代方案：浅层整合（推荐）

放弃"迁移"思路，改为"数据贯通"：

```
步1：旧服务 /api/health 加 cron 运行状态（已基本完成，只是缺字段暴露）
步2：旧服务关键事件（新员工入职、薪资调整）→ 写一条消息到新服务的 agent_v2_cron_runs 表
步3：新服务的 Agent 可以查询旧服务的 DB 表（已共享同一库）
步4：逐步替换旧服务的前端入口（从 index.html 换成 agents-service-v2 的 admin.html）
```

这样可以：
- **零风险** — 不改变任何现有服务的通信方式
- **逐步整合** — 从数据共享开始，不涉及代码迁移
- **渐进式淘汰** — 旧服务最终只剩下 DB 查询功能，业务逻辑逐步由新服务的 Agent 接管

### 如果必须迁移（替换旧服务）

划分 3 个阶段：

```
阶段 A（低风险，2-3天）：
  纯查询 API → 考勤查询、薪资查询、审批列表
  这些只是 SELECT，不涉及事务，迁移到新服务加代理路由

阶段 B（中风险，1-2周）：
  纯写入 API → 日报提交、检查签到、积分记录
  需要验证旧服务不再写入时才切

阶段 C（高风险，2-4周）：
  有副作用的定时任务 → 飞书同步、报表生成、文件夹导入
  需要"先禁旧、再开新"的互斥策略
```

### 最终建议优先级

| 优先级 | 操作 | 理由 |
|--------|------|------|
| P0 | **修复旧服务的 DB 权限问题**（sales_raw 表 owner 不对） | 日志中已看到持续报错 |
| P1 | **修复旧服务的 `sendSystemAlert error: column "status" does not exist`** | 监控告警功能坏了 |
| P2 | **旧服务 /api/health 增加 cron 运行统计** | 完善可观测性 |
| P3 | **考虑将旧服务部分纯查询 API 迁移到 agents-service-v2** | 减少一个服务的复杂度 |
| P4 | **旧服务单文件 19,366 行 → 模块拆分** | 提高可维护性（但优先级低） |
| 不推荐 | 完整迁移旧服务到新服务 | 成本高、收益低、风险大 |
